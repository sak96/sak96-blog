Git Simplified (Part 1): Introduction

This article provides introduction to git internals.

Read time: 3 min
Written on April 29, 2020

Git is the stupid content tracker (as per it's man page). It lets you keep track of modification done to content. This gives a timeline of how content developed over time. Content is not tracked at file level, More explained in Linus Torvalds talk on git.

Content tracking may be required for various reasons.

  • Track Ownership of changes.
  • Backup and Restore to certain older point.
  • Synchronization between different sources (by agreeing on common point to sync with).

To learn how to use Git, you mostly likely will find guides explaining about git command line usage. Once you are good with command line usage, you may find it helpful to know about "how git works". This series of articles aim to familiarize reader with high level over view of git working. Series combines articles describing independent topic. This article introduction to git internals.

Introduction

Git like other content tracking system functions by recording state of content at given time. Each such state are called as commits. The whole tracking system is Directed Acyclic Graph of commits. Each commit is identified by a hash.

Directed Acyclic Graph

  • Example: child -> parent -> grandparent
  • Graph is structure which represents relation between objects using connection.
    • The object in this case is commits.
    • The relationship in this case is parent-child relationship.
    • Multiple child can share single parent.
    • Single child may be created by Multiple parents.
  • Directed refers to the fact that these relationships have a direction.
    • The direction in this case is from child to parent.
    • The arrow head points to the direction.
  • Acyclic means the graph does not have cycles or loops.
    • A commit or its parents (recursively) cannot point back to the commit itself.
  • Better explained by wiki.

Hash

  • Hash function are function whose input is variable size and output is fixed size.
  • Hash of data is output of hash function with the data as input.
  • Cryptographic hashes are hash functions which are hard to reverse. In simple terms, for given hash(output) the input is hard to determine.
  • Hashing function used in git is sha-1 (or sha-256 in recent versions. Check hash function transition).

Data Model

Git data model can be at high level categorized into two variations.

  • Object store: Map of hash to objects. Ex: commit
  • References: Maps human-readable strings to hashes. Ex: branches
